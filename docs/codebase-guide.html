extractorPie / codebase manual

Every file, what it does, and how to rebuild it

A walk through the whole codebase, back to front. Read the big picture and the three core patterns first; after that, each file's section explains its purpose, how it works, and what you would need to know to write it yourself.

The big picture

Understand this and every file has an obvious place.

extractorPie is two halves that meet at one JSON endpoint. The backend (Python, FastAPI) knows who you are, remembers what you follow, and on request goes out to a dozen third-party APIs, cleans up their wildly different responses into one shape, and hands the frontend a single tidy object. The frontend (plain HTML, CSS, and one JavaScript file) is a small single-page app that logs you in, walks you through onboarding, and renders that object as a dashboard and tabs.

There is no framework on the frontend and no ORM magic you cannot see. Every moving part is a file you can read top to bottom. The layering, from the outside in:

frontend/ the browser app (auth, onboarding, dashboard) | HTTP + a bearer token api.py routes: /auth, /me, /search, /feed | feed.py composes one user's feed from their interests | runs many adapters at once adapters/ one file per source, each returns normalized items | guarded by cache.py + per-source failure isolation (base.py) db.py Postgres: users, interests, follows security.py password hashing + signed session tokens config.py reads .env: secret, db url, API keys

The request lifecycle (what happens when the dashboard loads)

  1. The browser has a saved token. It calls GET /api/feed with Authorization: Bearer <token>.
  2. api.py validates the token (security.py), loads that User and their Interest and Follow rows (db.py).
  3. It calls build_feed(interests, follows) in feed.py.
  4. feed.py looks at which domains the user enabled and fires the matching adapters concurrently. Each adapter call is wrapped by run_adapter (base.py), which checks the cache first and swallows any error into an "errors" list instead of crashing.
  5. Each adapter returns a list of items in the one normalized shape. feed.py sorts and buckets them into a per-domain payload (events vs news, sub-tabs for Screen).
  6. The JSON comes back; app.js renders the dashboard and tabs from it.

That is the entire system. The rest of this manual is just each of those boxes in detail.

Three core patterns

These recur everywhere. Learn them once here.

1. The normalized item shape (the contract)

Every source returns data in a different format. Rather than let that mess leak into the frontend, every adapter converts its data into one dictionary shape. The frontend only ever knows about this shape, so adding a new source never changes the frontend.

{
  "source":   "Hacker News",   # where it came from (shown as a label)
  "domain":   "tech",          # tech | sports | games | screen
  "kind":     "news",          # "news" (feed item) or "event" (dated, calendar)
  "title":    "...",
  "subtitle": "...",           # secondary line: outlet, score, episode
  "url":      "...",
  "ts":       "2026-07-12T14:16:00+00:00",  # ISO time; events must have one
  "group":    "anime"          # sub-bucket: movies/tv/anime, or a league
}

The distinction between kind: "news" and kind: "event" is the whole "streams vs follows" idea in one field. Events have a date and belong on a calendar; news flows and belongs in a feed.

2. The adapter pattern

Each source lives in its own file under adapters/ and exposes one async function that returns a list of normalized items. They all look the same from the outside, so feed.py can treat them interchangeably and you can add a source without touching anything else.

3. Cache-first with failure isolation

Every adapter call goes through run_adapter in base.py. It does two jobs: return a cached result if one is still fresh (so tight free-tier rate limits are respected), and if the adapter throws, record the message in an errors list and return an empty list instead of letting the exception bubble up. That is why one dead or unconfigured API shows a small error card while the rest of the page renders fine.

The one sentence to remember

Adapters normalize, the cache and run_adapter protect, feed.py composes, api.py exposes, the frontend renders. Every file is one of those five jobs.

Backend files

Python, FastAPI. Everything under backend/.

backend/requirements.txtdeps

The Python dependencies. Nothing exotic:

  • fastapi + uvicorn: the web framework and the server that runs it.
  • httpx: an async HTTP client, used by every adapter to call the third-party APIs.
  • sqlalchemy + psycopg: the database toolkit and the Postgres driver.
  • feedparser: parses RSS feeds (used by the news adapter).
  • python-dotenv: loads the .env file. email-validator: lets FastAPI validate the email field on signup.

Rebuild note

Install with pip install -r requirements.txt. Password hashing and tokens use only the Python standard library, so there is deliberately no crypto dependency here.

backend/.env.exampleconfig template

A template for the real .env (which is never committed). You copy it to .env and fill in values. It documents two required settings, SECRET_KEY and DATABASE_URL, and the optional API keys. The comments explain how to get each key and how to create the Postgres database.

Why it matters

Secrets live in .env, never in code. config.py reads them. The app runs with none of the API keys set, keyed modules simply show a "key not configured" card until you add them.

backend/app/config.pysettings

The single place that reads the environment. It calls load_dotenv(...) to pull in .env, then exposes each value as a module-level constant, with a sensible default where one makes sense.

from dotenv import load_dotenv
load_dotenv(Path(__file__).resolve().parent.parent / ".env")

SECRET_KEY   = os.getenv("SECRET_KEY", "dev-only-secret-change-me")
DATABASE_URL = os.getenv("DATABASE_URL", "sqlite:///./extractorpie.db")
TMDB_TOKEN   = os.getenv("TMDB_TOKEN", "")
# ... one line per key ...
TOKEN_TTL_SECONDS = 60 * 60 * 24 * 30   # sessions last 30 days
HTTP_TIMEOUT = 12.0                      # seconds, for every outbound API call

Rebuild note

The pattern is "read once, import everywhere." Any other file that needs a setting does from .config import TMDB_TOKEN. Defaulting DATABASE_URL to SQLite is what lets the app boot with zero setup; you override it with the Postgres URL in .env.

backend/app/db.pydata layer

Defines the database connection and the three tables using SQLAlchemy's modern typed style. This is the app's memory.

The engine and session

engine is the connection pool built from DATABASE_URL. SessionLocal is a factory for short-lived database sessions. get_session() is a FastAPI dependency that yields one session per request and always closes it.

The three models

  • User: id, email (unique), password_hash, created_at. Has relationships to its interests and follows, with cascade="all, delete-orphan" so deleting a user cleans up their rows.
  • Interest: one row per enabled domain. Columns are user_id, domain (tech/sports/games/screen), and config, a JSON column holding that domain's refinement. The docstring gives the shapes, for example sports stores {"leagues": ["PL", "PD"]}, games stores {"esports": "valorant"}.
  • Follow: a followed item. kind is "tv" or "anime", external_id is that source's ID (a TVmaze or AniList id), and title is cached so the UI can show it without a lookup.

init_db() calls Base.metadata.create_all(engine), which creates any missing tables on startup.

Design choice

Storing each domain's refinement in one flexible config JSON column, rather than a rigid table per domain, is what lets the "streams vs follows" model stay simple. Sports keeps a league list, games keeps an esports flag, tech keeps sub-topics, all in the same column. Followed items (TV, anime) are the exception that gets its own Follow table because there can be many per user.

Rebuild note

The roadmap calls for Alembic migrations in a real build; create_all is the quick-start version that creates tables but does not evolve them. Fine until the schema starts changing.

backend/app/security.pyauth crypto

Password hashing and session tokens, using only the standard library so there is no crypto dependency to install.

Passwords

hash_password generates a random salt and runs PBKDF2-HMAC-SHA256 with 300,000 iterations, then stores salt$digest as one string. verify_password re-hashes the attempt with the stored salt and compares using hmac.compare_digest (a constant-time comparison that avoids timing leaks). Passwords are never stored or logged in plain text.

Tokens

A session token is userid.expiry.signature. create_token builds userid.expiry and appends an HMAC signature keyed by SECRET_KEY. verify_token recomputes the signature, checks it matches (again constant-time), and checks the expiry is in the future, returning the user id or None.

def create_token(user_id):
    expiry  = int(time.time()) + TOKEN_TTL_SECONDS
    payload = f"{user_id}.{expiry}"
    return f"{payload}.{_sign(payload)}"
Honest limitation

These are self-contained stateless tokens: there is no server-side session store, so you cannot revoke a single token before it expires (short of rotating SECRET_KEY, which logs everyone out). That is a fine trade for a personal app. For anything bigger you would move to a library like authlib or short-lived tokens plus refresh.

Rebuild note

The key insight is that a signed token needs no database lookup to validate, the signature proves it was issued by you and the expiry proves it is current. The secret must be long and random and must never ship in code.

backend/app/cache.pyttl cache

A tiny in-memory time-to-live cache: one module-level dict mapping a key to (expiry_time, value). get returns the value if it has not expired (deleting it if it has), put stores a value with a TTL.

Why it exists

Several free API tiers are strict: AniList allows 30 requests a minute, balldontlie 5, API-Football 100 a day. Without caching, a few page loads would hit the limit and start getting rejected. Caching makes the app call each API only on a cache miss.

Rebuild note

This is deliberately the simplest thing that works: one process, one dict. If the app ever runs as multiple worker processes, this cache would not be shared between them and you would swap it for Redis. The interface (get/put) is small precisely so that swap is easy.

backend/app/adapters/base.pyadapter contract

The heart of the adapter system. Three things live here.

The item() helper

A small factory that builds the normalized item dict with sensible defaults, so adapters do not repeat the key names. Its docstring is the canonical definition of the normalized shape.

run_adapter()

The wrapper every adapter call goes through. Its logic:

async def run_adapter(name, ttl, fetch, errors):
    cached = cache.get(name)
    if cached is not None:
        return cached                 # fresh cache: skip the network
    try:
        items = await asyncio.wait_for(fetch(), timeout=20)
        cache.put(name, items, ttl)
        return items
    except Exception as exc:      # isolation is the whole point
        errors[name] = str(exc)
        return []                     # empty, never an exception

This single function is why the app is resilient: a broken source becomes an entry in errors and an empty list, so feed.py keeps going and the frontend shows an error card for just that source.

KeyMissing

An exception the keyed adapters raise when their credential is not set. Because run_adapter catches all exceptions, a missing key naturally becomes a friendly error card whose message says exactly where to get the key.

Rebuild insight

If you rebuild only one file carefully, make it this one. The contract (item) plus the guard (run_adapter) are what make every other adapter short and every failure graceful.

backend/app/adapters/*.pythe 9 sources

Nine source files, all the same shape: an async function that fetches, then maps each result through item(...). They differ only in the API they call and how they translate it. Here is what each one does and its one interesting wrinkle. Full auth and rate-limit details for every one are in SOURCES.md.

hackernews.py — tech news, no key

Hacker News returns arrays of story IDs, not stories. So this adapter fetches the top-story IDs, then fans out one request per story concurrently with asyncio.gather, and maps each into a news item. The wrinkle: you always need that second round of requests to turn IDs into titles.

rss.py — generic news, no key

One adapter that serves every news lane, parameterized by a FEEDS table mapping each domain to its feed URLs. It fetches with httpx, then parses with feedparser (which is synchronous, so it runs in a thread via asyncio.to_thread). For the Screen domain, each feed carries a group of movies, tv, or anime, which is how the Screen news sub-tabs get filled. If every feed in a lane fails, it re-raises so the lane shows an error card.

tvmaze.py — TV shows, no key

Two functions: search_shows powers the follow flow during onboarding, and fetch_next_episodes takes the user's followed show IDs and asks TVmaze for each show with its next episode embedded. The wrinkle: a show between seasons has no next episode, so those return None and are filtered out, no fake countdown.

anilist.py — anime, no key

Uses AniList's GraphQL API (one POST endpoint, a query in the body). search_anime for onboarding, fetch_airing for the calendar. The wrinkle: it batches all followed anime into one query using id_in: $ids, which matters because AniList's rate limit is tight. Its nextAiringEpisode field gives an exact air timestamp, which is why it beats Jikan as the anime source.

tmdb.py — upcoming movies, keyed

Calls TMDB's upcoming-movies endpoint with the v4 bearer token. Raises KeyMissing if TMDB_TOKEN is unset. The wrinkle: results are region-sensitive, so it passes a region; movies become calendar events grouped under "movies".

football_data.py — soccer fixtures, keyed

Fetches fixtures for the user's chosen competitions in the next week using the cross-competition /matches endpoint (one call for all leagues, to respect the 10-per-minute limit). It also exports FREE_COMPETITIONS, the league list shown during onboarding.

balldontlie.py — NBA games, keyed

Fetches upcoming NBA games. The wrinkle is an auth quirk: the API key goes in the Authorization header raw, with no Bearer prefix. Free tier reaches games only, which is all this needs.

igdb.py — game releases, keyed via Twitch

The most involved adapter. IGDB authenticates through Twitch: it first exchanges a client id and secret for an access token (cached in a module dict until near expiry), then POSTs a query in IGDB's "Apicalypse" text syntax (not normal query params) to the release-dates endpoint. It dedupes multi-platform releases so a game appears once.

vlrgg.py — Valorant esports, self-hosted

Talks to a self-hosted vlrggapi instance (there is no official vlr.gg API). Raises KeyMissing unless VLRGG_BASE_URL is set. Deliberately best-effort: it is an unofficial scraper, so it is never allowed to be load-bearing. Returns matches with relative times in the subtitle rather than a precise timestamp.

The rebuild lesson across all nine

Every adapter is "call the API, loop the results, return item(...) for each." The differences are entirely in each API's auth and quirks. Once you have written two of them, the rest are variations. Start with hackernews.py and rss.py because they need no keys.

backend/app/feed.pyorchestrator

The composer, and the most important file to understand. It turns a user's interests and follows into the exact payload the frontend renders. Read this one closely; it is where all the decisions live.

What it holds

  • TTL: a dict of cache lifetimes per source, tuned to each API's cadence and rate limit (tech every 15 minutes, AniList every hour because its limit is tight).
  • SOURCE_DOMAIN: maps each source name to the domain whose error area its failures belong in.
  • build_feed(interests, follows): the async function that does the work.

How build_feed works

  1. Turn the interest rows into a domains dict and split follows into TV and anime lists.
  2. Open an asyncio.TaskGroup and, for each enabled domain, create tasks for its adapters. Every task is wrapped in run_adapter, so all of them run concurrently and none can crash the group. Keyed follow-based calls (TVmaze, AniList) use a cache key that includes the followed IDs, so the cache invalidates when you follow something new.
  3. After the group finishes, pull each task's result and assemble the payload: for each domain, an events list (sorted by date) and a news list (sorted newest first). Screen is special, its news is split into movies, tv, and anime sub-lists, and tech deliberately has no events (it is calendar-poor by design).
  4. Finally, turn the collected errors into a list of {source, domain, message} so the frontend can place each error card in the right tab.
async with asyncio.TaskGroup() as tg:
    if "tech" in domains:
        tasks["hn"]       = tg.create_task(gather_into("hackernews", ...))
        tasks["rss_tech"] = tg.create_task(gather_into("rss:tech", ...))
    if "screen" in domains:
        tasks["tmdb"] = tg.create_task(...)   # upcoming movies
        if tv_follows:   tasks["tvmaze"]  = tg.create_task(...)
        if anime_follows: tasks["anilist"] = tg.create_task(...)
    # ... games, sports ...
The two shaping rules encoded here

Screen's news is a dict of three sub-lists (the Movies/TV/Anime sub-tabs), and Tech gets no events list at all. Those two lines are the entire information-architecture decision from the roadmap, expressed in code.

Rebuild note

The shape to internalize: gather concurrently, isolate each call, then bucket into domains[x] = {events, news}. If you rebuild this, write it for one domain (tech) first, confirm the payload, then add the others one at a time.

backend/app/api.pyhttp routes

Every HTTP endpoint, grouped by concern. FastAPI turns each decorated function into a route and validates request bodies against the Pydantic models.

Auth

POST /auth/register and POST /auth/login take a Credentials model (email + password, with length validation). Register hashes the password and creates the user; login verifies it. Both return a token and whether the user has finished onboarding. current_user is a dependency that reads the Authorization header, verifies the token, and loads the user, raising 401 if anything is off. Any route that needs a logged-in user just adds user = Depends(current_user).

Profile and interests

GET /me returns the user's domains, follows, and the available leagues (for onboarding). PUT /me/interests replaces the user's interest rows from the posted domain-to-config map, validating that domains are known.

Follows and search

GET /search?kind=tv|anime&q=... proxies to the TVmaze or AniList search adapter. POST /me/follows and DELETE /me/follows/{id} add and remove followed shows, with an ownership check on delete.

Feed

GET /feed is a one-liner: it calls build_feed(user.interests, user.follows) and returns the result. All the work is in feed.py; the route just wires the current user to it.

Rebuild note

The pattern throughout is "validate input with a Pydantic model, get the user from the token dependency, touch the database, return JSON." FastAPI handles the validation and error responses; you write the intent.

backend/app/main.pyentry point

The application entry point that ties everything together. It creates the FastAPI app, adds CORS middleware (so the browser can call the API), includes the router from api.py, calls init_db() on startup, and mounts the frontend/ folder as static files so the backend serves the UI directly at /.

app = FastAPI(title="extractorPie")
app.add_middleware(CORSMiddleware, allow_origins=["*"], ...)  # dev; tighten later
app.include_router(router)

@app.on_event("startup")
def startup(): init_db()

app.mount("/", StaticFiles(directory=frontend_dir, html=True))

Rebuild note

Serving the frontend from the backend means one server and no CORS issues in production. In development the CORS wildcard lets a separate frontend dev server talk to it too. You run the whole thing with uvicorn app.main:app.

Frontend files

No framework. Three files under frontend/.

frontend/index.htmlshell

A near-empty HTML shell: a <div id="app"> with a loading message, the stylesheet, and app.js. Everything visible is drawn by JavaScript into that div. There is no server-side templating; the page is a blank canvas that app.js fills.

Rebuild note

This is the "single-page app" starting point: one root element, one script. The HTML barely changes after this.

frontend/style.cssstyling

All the styling, organized by the views it supports: CSS variables for the palette at the top, then the auth/onboarding cards, then the main app shell (sticky header, tabs), then the content pieces (agenda rows, news items, error cards, sub-tabs), then a mobile block at the bottom.

Worth knowing

The mobile media query is what fixes the header on small screens: it lets the header wrap and forces the tab nav onto its own full-width row (found and fixed during testing, the tabs were getting squeezed to nothing otherwise). Colors are all CSS variables, so re-theming is a matter of changing the :root block.

frontend/app.jsthe whole UI

The entire frontend behavior in one file: state, API calls, and the render functions for each view. It is long but flat, no framework, just functions that build HTML strings and wire up click handlers.

State

A single state object holds everything: the auth token (also mirrored to localStorage so you stay logged in), the loaded profile (me), the fetched feed, which tab and sub-tab are active, and scratch data for onboarding. There is no reactive framework, when state changes, the relevant render function is called again to redraw.

The api helper

call(path, options) is the one function that talks to the backend. It attaches the bearer token, parses JSON, throws on errors (surfacing the backend's message), and signs you out automatically on a 401.

The views (each a render function)

  • renderAuth: the sign-in / create-account card, toggling between the two modes.
  • renderOnboarding: the domain picker plus per-domain refinement, league chips for sports, the Valorant toggle for games, sub-topic chips for tech, and the search-and-follow flow for TV and anime. This is where the "different domains ask different questions" idea becomes UI.
  • renderMain: the app shell, the header with the tab nav, and it delegates the content to either the dashboard or a domain renderer.
  • renderDashboard: the unified agenda (all events across domains, sorted) plus the top two headlines per domain. It is explicitly a subset of the tabs, never its own data.
  • renderDomain: one tab's content, the calendar zone (skipped for tech) and the news feed. For Screen it renders the Movies/TV/Anime sub-tabs.

Helpers and boot

Small helpers format dates (dayLabel, timeLabel, agoLabel) and escape text to prevent HTML injection (esc). At the bottom, boot() runs on load: if there is a saved token it loads the profile and either shows the dashboard or onboarding; otherwise it shows the auth screen.

Rebuild insight

The mental model is: state in one object, one render function per screen, re-render on change. This is what frameworks like React automate; doing it by hand once makes clear what they are actually doing. If you rebuild the frontend, start with renderAuth, then renderDashboard against a fake feed object, and add the rest once data flows.

How to rebuild it yourself

The order that keeps you running at every step.

Do not build in file order; build in dependency order, so you have something working from the very first stage. Each row leaves you with a running app.

StepFilesYou can now...
1config.py, db.py, security.pyStore a user and issue a valid token (no routes yet, test in a REPL).
2api.py (auth only), main.pyRegister and log in over HTTP. Server boots and serves.
3adapters/base.py, cache.py, adapters/hackernews.pyFetch one real source into normalized items, cached and failure-isolated.
4feed.py (tech only), api.py (/feed)Return a composed feed for one domain over HTTP.
5frontend/ (index, style, app.js)See it in a browser: log in, view a tech feed.
6interests + follows in api.py, onboarding in app.jsPick domains and follow shows; feed reflects choices.
7remaining adapters + feed.py branchesAdd sources one at a time, each behind its cache and error card.
8dashboard + tabs + sub-tabs in app.jsThe full information architecture.
The single most important idea

The normalized item shape plus run_adapter are the spine. Every source bends to that shape, every failure is caught there, and feed.py only ever works with clean, uniform data. Get those right and the rest is straightforward wiring.
